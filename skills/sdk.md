## Three-tier development strategy

Route every operation to the right tool:

```
TIER 1 — now-sdk Fluent (metadata-as-code)
  Tables, flows, business rules, catalog items, ACLs, script includes
  TypeScript → build → deploy as scoped app
  ↓ falls through when runtime data ops or live queries needed

TIER 2 — Direct REST API (runtime operations)
  CRUD on live data, ad-hoc queries, update set management
  Use curl or SN REST API with admin credentials
  ↓ falls through when REST can't reach it (flow compile, UI Policy linking)

TIER 3 — Background Script (manual fallback)
  Run in Scripts–Background on the PDI (System Definition → Scripts - Background)
  GlideRecord operations, flow compilation triggers, bulk data ops
  Used for: anything neither Fluent nor REST can automate
```

| Operation | Tier |
|-----------|------|
| Create table + schema | 1 — `Table()` |
| Business rules | 1 — `BusinessRule()` |
| Flows | 1 — `Flow()` DSL |
| Catalog items + variable sets | 1 — `CatalogItem()` |
| ACLs, roles | 1 — `Acl()`, `Role()` |
| Script includes | 1 — `ScriptInclude()` |
| Seed / demo data | 1 — `Record()` with `installMethod` |
| Query live records | 2 — REST GET on table API |
| Update existing live records | 2 — REST PATCH |
| Bulk data loads | 3 — Background Script |
| Flow compilation after deploy | 3 — Background Script or Flow Designer UI |
| UI Policy action linking | 3 — Background Script |

## App creation and first deploy

```bash
# Creates app on PDI AND scaffolds locally in one step
snc init \
  --appName    "My App" \
  --packageName "my-app" \
  --scopeName  "x_<VENDOR_PREFIX>_my_app" \
  --template   "typescript.basic" \
  --auth       <YOUR_INSTANCE>

npm install
npm run build && npm run deploy
```

`snc init --auth <profile>` registers the app on the PDI. Subsequent `npm run deploy` calls update it. If scope in `now.config.json` has no matching app on the PDI → `application was null`.

## Key helpers

| Helper | Purpose | Example |
|--------|---------|---------|
| `Now.ID['key']` | Stable sys_id reference, auto-generated on first build | `$id: Now.ID['my-rule']` |
| `Now.include('./file.js')` | Embed external script (path relative to `.now.ts` file) | `script: Now.include('./rule.server.js')` |
| `Now.ref(export)` | Cross-reference another Fluent export | `table: Now.ref(x_<prefix>_<app>_sn_release)` |
| `Now.attach('./file')` | Attach a file to a record | `attachment: Now.attach('./logo.png')` |
| `TemplateValue({})` | Set field values in flow update/create actions | `values: TemplateValue({ state: '2' })` |
| `wfa.dataPill(ref, type)` | Reference a flow data pill | `wfa.dataPill(params.trigger.current.sys_id, 'reference')` |

## Seed data — Record() with installMethod

```typescript
import { Record } from '@servicenow/sdk/core'

Record({
    $id: Now.ID['seed-my-record'],
    $meta: { installMethod: 'first install' },   // 'demo' for demo-only data
    table: 'x_<prefix>_<app>_my_table',
    data: {
        name: 'My Value',
        active: true,
    },
})
```

- `'first install'` → packed into `unload/` — installed on every deploy, idempotent by `$id`
- `'demo'` → packed into `unload.demo/` — only when demo data is requested
- Omit `$meta` → always installed

Do **not** use Fix Scripts for seed data — `Record()` is idempotent and version-controlled.

## Flows — DSL patterns

```typescript
import { Flow, action, wfa, trigger } from '@servicenow/sdk/automation'

export const myFlow = Flow(
    {
        $id: Now.ID['my-flow'],
        name: 'My Flow',
        description: 'What this flow does',
    },
    wfa.trigger(
        trigger.record.created,
        { $id: Now.ID['my-flow-trigger'] },
        {
            table: 'x_<prefix>_<app>_my_table',
            condition: 'active=true',
            run_flow_in: 'background',
            run_on_extended: 'false',
            run_when_setting: 'both',
            run_when_user_setting: 'any',
            run_when_user_list: [],
        }
    ),
    (params) => {
        // Look up a record
        const found = wfa.action(
            action.core.lookUpRecord,
            { $id: Now.ID['my-flow-lookup'] },
            {
                table: 'sys_user',
                conditions: 'active=true^email=admin@example.com',
                if_multiple_records_are_found_action: 'use_first_record',
            }
        )

        // Update a record
        wfa.action(
            action.core.updateRecord,
            { $id: Now.ID['my-flow-update'] },
            {
                table_name: 'x_<prefix>_<app>_my_table',
                record: wfa.dataPill(params.trigger.current.sys_id, 'reference'),
                values: TemplateValue({ assigned_to: wfa.dataPill(found.Record.sys_id, 'reference') }),
            }
        )
    }
)
```

### Flow triggers

| Trigger | Fires when |
|---------|-----------|
| `trigger.record.created` | Record inserted |
| `trigger.record.updated` | Record updated |
| `trigger.record.createdOrUpdated` | Either |
| `trigger.record.deleted` | Record deleted |

### Flow actions (action.core.*)

| Action | Purpose |
|--------|---------|
| `action.core.log` | Log with level (info / warn / error) |
| `action.core.lookUpRecord` | Query for single record |
| `action.core.createRecord` | Insert a new record |
| `action.core.updateRecord` | Update existing record |
| `action.core.sendEmail` | Send email notification |
| `action.core.sendNotification` | Send via notification template |

## Error reference

| Error | Cause | Fix |
|-------|-------|-----|
| `application was null` | Scope in `now.config.json` has no matching app on PDI | Re-run `snc init --scopeName x_<VENDOR_PREFIX>_<app> --auth <YOUR_INSTANCE>` |
| `Could not find package.json` | Wrong working directory | Run from repo root (`axiom-lite/`) |
| `Invalid scope` | Wrong vendor prefix | Prefix for <YOUR_INSTANCE> is **9274** |
| `Cannot find module '@servicenow/sdk'` | Missing deps or wrong SDK path | `npm install`, then verify `snc --version` |
| `fetch failed` on `--reinstall` | Network/auth issue | Retry without `--reinstall`; check `snc auth --list` |
| `Custom table not in trigger types` | Stale type defs | `npm run types` after new table deploy, then rebuild |
| `active is not valid in FlowConfigProps` | Unsupported field | Remove `active` — activate in Flow Designer after deploy |
| `must be inside wfa.dataPill call` | Direct field access in flow | Wrap: `wfa.dataPill(expr, 'type')` |
| `Flow import from wrong module` | Imported from `/core` | Import `Flow`, `action`, `wfa`, `trigger` from `@servicenow/sdk/automation` |
